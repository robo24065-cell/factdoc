# FactDoc (가칭) — 국가 공식 데이터 기반 건강정보 팩트체커

> **한 줄 정의**: TV·홈쇼핑·유튜브 등에서 퍼지는 건강 주장을, 질병관리청(+식약처)의 국가 공식 데이터에 근거해 **사실 / 부분적·과장 / 근거없음·허위 / 공식근거 없음(보류)** 4단계로 판정하고, 출처와 근거 체인을 함께 보여주는 AI 서비스.
> **출품 대회**: 2026 병무청·방위사업청·질병관리청 합동 공공데이터·AI 활용 경진대회 — **제품 및 서비스 개발 부문** (작동하는 시제품 필수).

**가드레일 요약 (절대 위반 금지 — 자세한 건 §10):**
- LLM이 생성한 답을 검증 없이 "정답"으로 캐시·제공하지 않는다. 모든 판정에는 공식 출처 인용 + 신뢰도 + 등급(자동·미검증 / 검증완료)이 붙는다.

- 의료 진단이 아님을 명시한다.

## 2. 제품 개요 & 가치

- **문제**: 쇼닥터, 홈쇼핑 건강식품 인포머셜, 유튜브 건강영상의 과대·허위 건강 주장. 특히 고령층이 표적이며 실제 건강 피해로 이어짐.
- **해결**: 국가가 보증하는 "잠자는" 공식 건강 데이터를 깨워, 건강 주장을 즉시 검증·반증.
- **사용자**: (B2C) 일반/고령 소비자 · (B2G) 식약처 부당광고·방심위 방송 모니터링 보조 · (B2B) 플랫폼 콘텐츠 신뢰도, 보험·약국·시니어케어.
- **서사(피칭 키)**: "국가 공식 의학정보(질병청)를 AI로 깨워 가짜 건강정보를 잡는다." — 비정형/저활용 공공데이터를 제품화한 사례.

## 3. 데이터 전략 (★질병청이 코어, 식약처는 보조)

> 원칙: **위험한 건강 주장의 대부분은 "질병" 주장**(치료·예방·감염병·만성질환)이고, 그 권위 데이터는 질병청 국가건강정보포털이다. 식약처는 "건강기능식품 광고"라는 한 카테고리만 담당한다. → 질병청이 자연스럽게 척추가 되도록 설계한다. (식약처가 주가 되면 안 됨.)

**코어 — 질병관리청 (필수 주최기관 데이터):**
- 국가건강정보포털 `health.kdca.go.kr` — 질병별 원인·예방·치료·관리의 국가 공식 의학정보 (= 검증 근거 엔진의 1차 인덱스)
- 만성질환건강통계 `chs.kdca.go.kr/cdhs` — 유병률·위험요인
- 감염병포털 `dportal.kdca.go.kr` — 감염병·백신 관련 공식 정보/통계
- 국민건강영양조사 `knhanes.kdca.go.kr` — 통계적 사실 근거 (원시자료 무료 공개)

**보조 모듈 — 식약처 (건강기능식품 주장 한정):**
- 건강기능식품 개별인정형 정보 / 기능성 원료인정 현황 / 건강기능식품정보 / 영양DB (공공데이터포털 `data.go.kr`, 식의약 데이터 포털 `data.mfds.go.kr`, 식품안전나라 `foodsafetykorea.go.kr`)
- **법적 룰(룰베이스 백본)**: 건강기능식품의 기능성은 질병의 직접 치료·예방이 아니며, 질병 치료·예방 효과를 표방하는 원료·성분은 인정 불가(기능성은 질병위험 감소·생리활성·영양소 기능 3종 한정). → "이 보조제가 ○○병을 치료/예방한다"류는 자동으로 과대/허위.

**병무청 데이터 주의**: 개인 단위 병역판정검사 기록은 API·다운로드로 제공되지 않음(집계 통계와 키·체중·BMI 정도만 공개). 풍부한 개인 병리 데이터(약 295만 건)는 데이터안심구역 `dsz.kdata.or.kr` 내에서만 분석되고 원자료 반출 불가. → 본 제품은 병무청 데이터에 의존하지 않는다(필요 시 집계 통계만 보조 활용).

## 4. 아키텍처 (전체 인프라 — 목표 형태)

```
[사용자 질문/주장 텍스트(또는 자막)]
        │
   ① 주장 추출·정규화 (작은 LLM)  →  canonical claim (주체·관계·대상질환)
        │
   ② 시맨틱 캐시 매칭 (벡터 ANN + 메타데이터 필터)
        ├─ 신선 히트 → 캐시된 판정 + 출처 요약 (빠름/저렴)
        ├─ 낡은 히트 → 즉시 응답 + 백그라운드 재검증 (stale-while-revalidate)
        └─ 미스 ↓
   ③ 검색 (하이브리드: BM25 + dense 벡터) + 리랭커  →  소유 코퍼스에서 근거 수집
        │
   ④ 판정 엔진 (★자체 방법): 룰 + 클레임 그래프 트리플 매칭, LLM은 가장자리에서만
        │
   결과: 4단계 판정 + 근거 체인 + 출처 인용 + 신뢰도
        │
   ⑤ 캐시에 저장 (출처·신뢰도·source-version·TTL·등급[자동/검증완료]·조회수)
```

**저장/지식베이스 레이어:**
- 소유 코퍼스: 질병청·식약처 공식 자료를 **벌크 다운로드해 로컬 소유**(호출당 외부 API 안 씀). 청크 + 임베딩. **변경분만 diff로 증분 갱신.**
- 캐시 항목 스키마: `{ canonical_claim, verdict(4단계), citations[], confidence, source_version, ttl, tier(자동·미검증|검증완료), query_count }`
- 인덱싱: 의미 매칭 = **벡터 ANN(pgvector)**, 필터 = 질환·주장유형·출처 메타데이터 인덱스, 라우팅·설명 = "질환 대분류→세부→주장유형" 분류 트리(온톨로지).

**백그라운드 잡 (사용량 적은 시간대 / 새벽):**
- 바뀐 공식 문서만 증분 인덱싱 (전체 리로드 금지).
- 당일 질문 로그에서 핫토픽 군집화 → 사전검증 후보 승격 (사람 검토 큐 경유).
- 트래픽 높은 + 휘발성 높은 항목만 갱신.
- **TTL은 주장 유형별로 차등**: 만성질환 사실=길게, 감염병·백신·유행·리콜=짧게.
- **출처 변경 기반 무효화**: 식약처 부당광고 적발·행정처분, 질병청 업데이트 발생 시 영향받는 캐시 자동 무효화 (시간 기반보다 우선).

**채택한 추가 기능 (모두 구현 — 제품 차별점):**
- **트렌드 레이더 + 주간 가짜정보 TOP 10 대시보드**: 질문 로그(+선택적 외부 트렌드)를 군집화해 내일 터질 가짜정보를 선제 검증. B2G·언론용 산출물 + 홍보 자석 + 발표 임팩트.
- **출처 변경 자동 감시·무효화기**.
- **카톡 공유카드**: "이 주장은 공식 근거상 ~ [출처]" 포워드 가능한 카드 (고령층 확산).
- **평가 플라이휠**: 실제 질문 로그가 곧 라벨링 테스트셋이 됨 → 캐시가 쌓일수록 §8의 검증 자료가 두꺼워짐.

## 5. 기술 스택 & 무료 배포 맵

**핵심 원칙: 바퀴는 사서 쓰고, 독창성 예산은 §6(자체 방법 2개)에만 쓴다.**

### 사서/조립해 쓸 것 (만들지 말 것)
- **DB/벡터/인증/스토리지/함수**: **Supabase** (Postgres + **pgvector** + Auth + Storage + Edge Functions, 넉넉한 무료 티어). → 관계형 + 벡터 + 전문검색 + 클레임 그래프를 한 곳에서. *Firebase(Firestore)는 NoSQL이라 클레임 그래프/관계형·벡터 요구에 부적합 → 데이터 레이어는 Supabase 권장. (원하면 Hosting/Auth만 Firebase 병행 가능.)*
- **프론트엔드 호스팅**: Cloudflare Pages 또는 Firebase Hosting (무료).
- **API/엣지 + 스케줄러**: Cloudflare Workers + **Workers Cron Triggers**(새벽 배치 무료). 시간제한 큰 무거운 배치(코퍼스 대량 갱신)는 **GitHub Actions cron** 사용.
- **레포/CI**: GitHub + GitHub Actions.
- **임베딩**: 한국어 되는 기성 모델(BGE-m3, multilingual-e5 류). 직접 학습 X. *착수 시점에 최신 한국어 임베딩만 확인해 채택.*
- **검색**: 하이브리드(BM25/전문검색 + dense 벡터) + 리랭커(cross-encoder). RAG 정석.
- **LLM**: 추출·매칭은 작고 싼 모델, 검증·설명만 큰 모델(티어링). *유일하게 비용 날 수 있는 부분 → 캐시 + 작은 모델로 최소화. (Gemini 무료 티어 등 활용 검토.)*
- **핫 캐시**: Upstash Redis(무료 티어).
- **STT(영상 스트레치)**: Whisper 류.
- **평가**: sklearn 지표 + 손으로 짠 평가셋. (faithfulness는 RAGAS 참고하되 팩트체커는 자체 평가가 더 믿음직.)

### 비용 메모
거의 무료. LLM 호출만 약간 비용 → 캐시 적중률 + 티어링으로 억제. API 키(Anthropic/OpenAI/Gemini)는 환경변수로 관리, 절대 레포에 커밋 금지.

## 6. 차별화 (★독창성 = 여기 2개에 집중)

> "Supabase + LLM + 스케줄러 조합"은 차별화가 아니다(다들 함). 진짜 독창성은 (1) 데이터 구조화·연결, (2) 자체 판정 방법 — 두 군데서 나온다.

### ① 검증용 데이터 구조 = 최대 해자
- 문단을 그냥 청크·임베딩(흔한 RAG)하지 말고, 공식 문서를 **구조화된 주장-근거 단위**로 추출한다.
- 스키마(예): `(주체, 관계[치료/예방/위험↑/효과없음/근거불충분], 대상질환, 근거수준, 출처, 한정조건)`
  - 예: "당뇨 관리" 글 → `(인슐린, 치료적관리, 제2형당뇨, 표준치료, 출처)`, `(특정 민간요법, 근거불충분, 당뇨, …)` → **클레임 그래프**.
- 식약처 인정기능성을 **룰 테이블**로 얹음: `(원료 → 인정 기능성, 질병치료표방=불가)`.
- **엔티티 정합 / 한국어 건강-클레임 온톨로지(동의어층)**: 홍삼=홍삼농축액=red ginseng, 당뇨=당뇨병=제2형당뇨=혈당 … 식약처↔질병청↔사용자 어휘 매핑.
- 손이 많이 가서 못 베끼는 자산 = 모트.

### ② 판정 엔진 = 두 번째 자체 방법
- "LLM에게 참이냐 묻기"가 아니라: **룰**(식약처 법규 등) + **클레임 그래프의 (주체·관계·대상) 트리플 매칭**으로 판정.
- LLM은 가장자리에서만: (a) 지저분한 사용자 주장 → 트리플 파싱, (b) 결과 설명 문장 생성. **진실 판단은 LLM이 하지 않는다.**
- 장점: 더 정확 + **왜 그렇게 판정했는지 근거 체인이 보임** → §8(AI 검증 30점)에 직결, 환각 방지.

(트렌드 레이더·출처변경 무효화·카톡카드·평가 플라이휠은 *제품 차별점*으로 포지셔닝. 기술 독창성의 중심은 ①②.)

## 7. 개발 로드맵 (★수직 슬라이스 우선 — 마감 7/7 인지)

> 원칙: 수평으로 층층이 쌓지 말고, **하나의 주장 카테고리를 끝까지 작동**시킨 뒤 그 위에 복제·확장한다. 매 Phase 끝에 항상 데모 가능한 상태를 유지한다.

- **Phase 1 — 수직 슬라이스 (E2E 1줄)**: 한 카테고리(예: "당뇨 완치 사기")만.
  - 관련 질병청 문서 수집 → 그 슬라이스용 클레임 그래프(소량, 반자동/수동 OK) → 판정 엔진 최소 구현 → 최소 웹 UI(주장 입력 → 4단계 판정 + 근거 체인 + 출처). Supabase 연결.
- **Phase 2 — 확장**: 카테고리 추가(감염병·백신, 만성질환 관리, 암 예방 등) + 식약처 건강기능식품 보조 모듈 + 시맨틱 캐시(벡터 매칭) 도입.
- **Phase 3 — 인프라**: 백그라운드 잡(증분 인덱싱·핫토픽·TTL), 출처변경 감시·무효화, 트렌드 레이더 + 주간 TOP 10 대시보드, 사람 검토 큐.
- **Phase 4 — 마무리/검증/피칭**: 스트레치(STT·브라우저 확장·카톡카드) 중 가능한 것, **평가 하니스 + 벤치마크 표(무근거 LLM vs 우리 시스템)**, 발표자료·사업화 슬라이드.

각 Phase는 §8의 점수와 연결되게 진행. 마감까지 못 갈 것 같으면 **Phase 1~2 완성 + 작동 데모**를 최우선, 인프라는 발전가능성 슬라이드로 보여준다.

## 13. ⭐ 확정 설계 (2026-06-14 세션) — 최신 합의 (이 섹션이 §5·§6·§7·§8을 구체화/우선)

> 멀티에이전트 검증(경쟁/독창성·데이터 라이선스·심사 시뮬 3인 패널·적대적 검증·아이디어 선별)을 거쳐 확정. 목표 = **"심사가 깎을 구멍을 0개로" → 빈틈 없는 80점대 후반 = 대상 경쟁권.** 96 같은 숫자를 기능으로 좇지 않음(초점 분산은 독창성·공공데이터 감점). 시뮬 기준선 ~70(우수) → 누수 메우면 공공데이터 27+/AI 26+/독창성 17+/발전 17+ ≈ 85±.

### 13.0 포지셔닝 (헤드라인 고정)
- **한 줄:** *"LLM은 진실을 판단하지 않는다 — 국가 공식데이터의 룰 + 클레임그래프가 판정한다."* (발표 첫 화면·신청서 요약·데모 상단 고정)
- **"RAG"는 헤드라인에서 제거** — 내부 검색 부품 이름으로만. (작년 이 대회 대상작이 'RAG 챗봇'이라 "또 RAG"로 1초 분류될 위험이 최대 실점 리스크.)
- 질병청 코어 전면, 식약처는 명시적 **보조**(건기식 룰 한정). "식약처 업무 래퍼"·"포털 단순검색" 오해를 선제 차단하는 한 줄을 데모·슬라이드에 삽입.

### 13.1 폭 전략 — '브레드스 엔진' + 검증 티어
- 질병을 손으로 늘리지 않는다. **자동 추출 파이프라인**(공식문서→청크→트리플 추출→룰 검증→적재)이 폭을 만든다. 질병 추가 = 파이프라인 1회 재실행. 폭 = 데이터 볼륨.
- **검증 티어:** 자동분 전부 `자동·미검증` 배지. 데모+평가 카테고리만 사람 스팟체크 후 `검증완료` 승격. → "수백 질환 커버(미검증) + 검증완료 코어" = 정직·확장 동시 서사.
- **초집중은 자동화 안 되는 4개 모트에만:** ① 룰+그래프 판정엔진 ② 한국어 온톨로지 ③ 식약처 룰테이블 ④ 평가 하니스. 코어가 전 카테고리에 재사용 → 폭이 거의 공짜.
- 폭+깊이는 트레이드오프 아님: **깊이를 한 번 깊게 파면 폭은 그 위에서 자동 복제.**

### 13.2 근거수준 = LLM이 아니라 '출처'가 결정 (★공공데이터 30점 신뢰 장치)
LLM은 트리플 *모양*만 파싱. `근거수준`·진실판단은 **결정론적 룰**이 한다 — "근거수준 누가 매겼나"에 "국가 출처 종류가 자동 결정"으로 답.

| 출처 | 부여 근거수준 |
|---|---|
| 국가건강정보포털 표준치료·권고 문단 | `공식권고/표준치료` |
| KNHANES·만성질환통계 수치 | `통계근거` |
| 식약처 인정기능성 레코드 | `인정기능성` |
| 식약처 부당광고 적발/행정처분 | `반증·규제근거` |
| 코퍼스 미수록 | `근거없음` → 보류 |

### 13.3 4단계 판정 루브릭 (경계 확정)
판정 대상은 항상 **주장(claim)**, 사람·브랜드 아님(명예훼손 회피).
- **사실**: 핵심 트리플이 공식 근거와 방향·강도·한정조건 모두 일치. (상향 금지: 일부만 뒷받침이면 '사실'로 안 올림.)
- **부분적·과장**: 근거에 핵심(seed of truth) 존재하나 강도 과장/범위 과장/한정조건 누락/인과 비약. *방향은 맞고 정도·조건이 틀림.*
- **근거없음·허위**: 핵심 트리플이 공식 근거와 정면 모순/근거가 명시 부정·경고/만성·비가역 질환의 완치·특효/식품의 질병 치료·예방 표방(식약처 룰).
- **공식근거없음·보류**: 대조할 공식 근거가 코퍼스에 없음 → 1급 결과. (시스템 실패 아님.)
- **경계 테스트(운영화):** ① 방향성(근거가 같은 방향 일부 지지=과장 / 침묵·반대=허위·보류) ② 핵심 트리플 보존(핵심 관계 존재=과장 / 부정·부재=허위·보류) ③ 완치 룰(비가역 만성질환 cures=원칙 허위) ④ 식약처 룰(식품 질병치료 표방=자동 허위). **허위 vs 보류:** 근거가 *있어서 부정*=허위 / *없어서 모름*=보류. 반증이 있을 때만 허위.

### 13.4 클레임 추출 스키마 + 관계 어휘
- 필드: `subject`(정규화 엔티티; 사람·브랜드 금지, 발화자는 speaker 메타로만) · `relation`(닫힌 집합) · `object_disease`(정규화 질환) · `polarity`(assert/negate) · `strength`(absolute/strong/moderate/weak) · `qualifier`(한정조건) · `evidence_level`(근거측에서 채움, §13.2) · `source[]`(citation; 모든 판정 1개+ 필수, 보류는 []) · `verdict`(4단계, tier+confidence 동반).
- **relation 닫힌 집합:** `cures / prevents / reduces_risk / increases_risk / manages / no_effect / insufficient_evidence / causes_or_worsens / diagnoses / replaces_treatment`.
- 파싱 프롬프트 규칙(요지): 진실판단 금지·의학지식 보태기 금지·환각 금지, 한 문장에 주장 여러 개면 트리플 분리, relation은 닫힌 집합에서만, subject에 사람/브랜드 금지, JSON만 출력. (전체 프롬프트·5개 변환예시는 코드 `prompts/`에 둠.)

### 13.5 온톨로지 시드 (당뇨 슬라이스 즉시 사용)
- 동의어 예: `제2형당뇨 ← 당뇨/당뇨병/성인당뇨/혈당병` · `홍삼 ← 홍삼농축액/홍삼정/red ginseng` · `혈당조절 ← 공복혈당/HbA1c/식후혈당` · `운동요법 ← 걷기운동/유산소운동/신체활동` · 민간요법 `여주/돼지감자/바나바잎`(코퍼스 미수록 시 보류 후보·사람검토 큐 태깅).
- 주장유형 카테고리: 완치·치료형(고위험) / 대체치료형(약·병원 대체, 고위험) / 예방·면역형 / 위험경고형(백신유해) / 건기식 질병효능형(식약처 룰 자동) / 통계인용형(KNHANES 대조) / 민간요법형(보류 후보).

### 13.6 평가 하니스 (★AI 30점 — 약속을 측정값으로)
- **라벨셋:** 6~8 질환군 × 30~40 멘트 = 250~300건(실제 홈쇼핑/유튜브/커뮤니티 멘트 + 합성 보강).
- **자가채점 차단:** 각 골드라벨에 **공식 출처 ID 부착**(라벨이 팀 의견 아니라 "어느 공식문서가 뭐라 했나"에 매임). **2인 독립 + 코헨 카파(목표 ≥0.6)**, 불일치 3자 조정.
- **지표:** 4클래스 정확도 + 클래스별 P/R/F1 + **인용 정확도**(인용 출처에 실제 근거 존재) + **환각률** + 보류 적정성 + 캘리브레이션.
- **confidence 산출식:** 룰·그래프는 확률 없음 → `confidence = f(매칭강도, 근거수준 가중, 트리플 커버리지, 검색점수)` 결정론적 점수 → reliability diagram. 프레이밍: "확률 환각 대신 규칙기반 검증가능 신뢰도".
- **간판 비교표(ablation):** (a)무근거 LLM (b)일반 RAG (c)RAG+룰 (d)풀(룰+그래프) + 컴포넌트별(파싱 F1·엔티티링킹·검색 recall@k). **베이스라인 = 한국어 무근거 LLM 59.8%**(SNU 500건 벤치마크 논문 인용).

### 13.9 추가 채택 기능 5개 (전부 같은 엔진·같은 질병청 데이터 — 초점 유지)
1. **판정 결정트리 (Why-Trace)** [low/★최우선] — 룰→트리플→근거수준 발화 경로를 LLM 설명과 분리해 시각화. 독창성+AI설명가능성. *단 하나만 추가한다면 이것.*
2. **유병률 정합성 검사기** [med] — 수치 주장을 KNHANES 연령·성별 유병률/관리율 분포와 대조해 통계적 outlier 자동 반증. 공공데이터+AI.
3. **근거 하이라이트 추적 (Span Grounding)** [med] — 근거 원문 문장·통계표를 span 하이라이트 + 인용 정확도 채점. AI(인용정확도·환각률)+독창성.
4. **근거수준 자동 등급화** [low] — §13.2 룰로 `evidence_level` 자동 산출.
5. **근거 커버리지 라우터** [low] — KDCA 커버리지 맵으로 사각지대 판별 → 보류를 데이터 기반으로 정당화. AI(환각억제)+공공데이터.
- **★ W1에 받침 3개 미리 심기(이게 위 기능들을 싸게 만듦):** ① 판정엔진이 '발화 로그' 남기게 설계(→Why-Trace는 UI만) ② 청킹 단계에서 `source_span` 저장(→Span Grounding은 하이라이트만) ③ 온톨로지 인덱싱하며 커버리지 맵 구축(→보류 라우터 공짜).

### 13.10 UI/UX — 대시보드(반응형 3종)·마이페이지·비만
- **대시보드 — PC / 태블릿 / 모바일 3개 화면을 각각 별도 레이아웃으로 설계·구현(반응형 분기 필수).** 폼팩터별 정보밀도·내비게이션·차트 배치를 따로 최적화(데스크톱=다패널 대시보드, 태블릿=2열 재배치, 모바일=세로 스택·핵심 카드 우선). 패널 5종: ① 클레임그래프 시각화(=구조 가시화) ② 트렌드 레이더+주간 가짜정보 TOP10(B2G·언론) ③ 판정 분포·인용정확도·환각률(평가 수치 가시화) ④ 질병청 통계 viz(연령/성별 유병률 등) ⑤ 🚨 실시간 유행/주의 감염병 트렌드(감염병포털 API — §13.10a).
- **반응형 원칙은 앱 전체(판정 UI·마이페이지 포함)에 적용**하되 대시보드는 3종 레이아웃을 명시적으로 분리.
- **마이페이지** — 선택 입력(연령대·성별·키·몸무게→BMI). 판정 결과를 **KNHANES 통계로 개인 맥락화**("50대 남성 당뇨 유병률 ~%, 표준관리는 ~ [출처]"). 가드레일: 선택 입력·LLM에 PII 미전송·인구통계 '맥락'이지 개인 진단 아님(의료면책).
- **비만/대사증후군을 정식 카테고리로 추가** — 질병청 영역 맞음(국가건강정보포털 비만 정보 + KNHANES 비만 유병률/복부비만 + 만성질환통계). 단 **골격근량·인바디는 질병청 공식통계 아님 → 핵심 엔진 제외, 발전가능성 B2B 제휴로만**.

### 13.10a 실시간 유행/주의 감염병 트렌드 패널 (대시보드 ⑤) — 감염병포털 연동
- **데이터:** 질병청 감염병포털 전수신고 감염병 발생현황 OpenAPI(data.go.kr 15139178, JSON/XML) → 현재 유행/급증 감염병 Top N + 추세. **per-request 외부호출 금지** — GitHub Actions cron(또는 pg_cron 트리거)으로 주기 수집해 Supabase에 캐시, 프론트는 캐시만 읽음. 유행성이라 **TTL 짧게**(§4 차등 TTL).
- **증상/예방 정보:** 국가건강정보포털 해당 감염병 문서 → '초기 증상'과 '예방·행동 수칙(손씻기·마스크 등)'을 **공식 인용 + 출처 링크 + 자체 요약**(변경금지 준수, 원문 통째 복제·재배포 금지).
- **클릭 플로우(핵심 — 팩트체커로 유입):** 핫이슈 질환 클릭 → ① 그 질환 관련 **시중 가짜뉴스 팩트체크 먼저**(판정 카드·근거체인) → ② 하단에 **"질병청 공식 진짜 증상·예방/행동 수칙" 카드**(출처 인용). = 트렌드 패널이 핵심 판정엔진으로 들어가는 디스커버리 퍼널.
- **가드레일:** 증상 정보는 **진단 아님·참고용**, 의심 시 전문가 상담 명시(§10.4 의료면책). 체크리스트는 '자가진단기'가 아니라 '질병청 안내 그대로'의 정보 카드. 가짜뉴스는 주장만 검증(발화자 불단정, §10.2).
- **점수 효과:** 감염병포털을 코어로 추가 사용 → 공공데이터 30 강화(주최기관 포털 다양화) + 시의성 공익 데모 = 발전가능성·심사장 한 방. §4 트렌드 레이더의 구체적 구현.
- **빌드 타이밍:** 감염병을 W2 질환군 확장에 포함 → 패널·클릭플로우는 W3. (실시간 무효화 같은 무거운 인프라는 범위 밖 — 배치 캐시로 충분.)

### 13.11 간판 데모 (독창성 20점 생사 — 3분할 비교)
같은 주장 1건을 **(무근거 LLM) / (일반 RAG) / (FactDoc 트리플 분해→룰 발동→근거체인+Why-Trace)** 3분할로 나란히. 간판 주장 3개(서로 다른 데이터 경로):
1. "당뇨는 △△즙으로 완치된다" → 질병청 표준치료 반증 + 완치룰 → **허위** *(질병청 코어)*
2. "○○건강기능식품이 당뇨를 치료한다" → 식약처 인정기능성 룰 자동 → **허위** *(식약처 룰)*
3. "신종 □□가 △△에 효과있다" → 코퍼스 미수록 → **보류** *(보류 1급화)*

### 13.12 독창성 프레이밍 (정직성=신뢰)
- 핵심 발상(클레임 트리플·KG 판정)은 해외 선행 존재(ClaimsKG'19, GraphCheck·SPO+UMLS'24, HealthFC) → 슬라이드에 **명시 인용**하고 "국제 연구는 PubMed·영어에 머물렀고, 우리는 **질병청 국가공식 의학정보를 한국어 온톨로지+식약처 법규 룰테이블로 결합한 첫 소비자 구현체**"로 선긋기.
- **정량 모트를 숫자로:** 트리플 N건 / 동의어 N쌍 / 식약처 룰 N항목 / 커버 질환 N개를 슬라이드에 박음("네이버/카카오가 왜 못 따라오나"의 답).

### 13.15 로드맵으로 강등(시제품 제외, 발전가능성 슬라이드로만)
입력채널 확장(홈쇼핑 OCR·유튜브 자막·카톡봇·보호자 알림·약국 위젯), 출력 재포장(제출용 PDF·공익 브리프·ROI 대시보드), 인프라(실시간 감염병 무효화·캘리브레이션 곡선·능동학습·반례 생성), 식단 사진분석/취향추천(저메추류), 골격근/인바디 코칭, 지역 맞춤맥락, 성분 상호작용. — 마감 리소스는 **엔진과 그 가시화에만**.
